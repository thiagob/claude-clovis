# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

`open-clovis` runs Claude Code as a persistent Docker container reachable via Telegram. It is a shell — infrastructure only. The actual work happens in a separate `clovis-workspace` repo mounted at `./data/workspace`.

Two-repo model:
- `open-clovis` — container, auth, Telegram config (this repo, managed from the host)
- `clovis-workspace` — git repo Claude operates on (mounted at `/home/clovis/workspace` inside the container)

## Key files

| File | Purpose |
|---|---|
| `Dockerfile` | Builds the image: node:22, Bun, Claude Code, gh CLI |
| `entrypoint.sh` | Registers plugin marketplace, installs Telegram plugin, configures git credentials, starts `claude` |
| `docker-compose.yml` | Two services: `agent` (Claude Code) + `n8n` (workflow automation), both on `clovis-net` |
| `setup.sh` | First-time setup: scaffolds `.env`, prompts for credentials, auto-generates `N8N_ENCRYPTION_KEY` |

## Common commands

```bash
./setup.sh                        # first-time init (prompts for credentials, generates N8N_ENCRYPTION_KEY)
docker compose build              # rebuild image after Dockerfile changes
docker compose run --rm agent     # interactive first-run wizard
docker compose up -d              # start headless
docker compose logs -f            # follow logs
```

## Volume layout

| Host path | Container path | Notes |
|---|---|---|
| `./data` | `/home/clovis` | Home dir — holds `.claude/` config and `.claude.json` |
| `./data/workspace` | `/home/clovis/workspace` | Workspace repo Claude operates on |
| `./n8n-data` | `/home/node/.n8n` (n8n container) | n8n workflows, credentials, settings |

`data/` and `n8n-data/` are gitignored. `.claude.json` is created automatically by `entrypoint.sh` on first start.

## Container internals

- Runs as user `clovis` (UID 1001). Host paths under `data/` must be owned by 1001.
- Claude Code is installed to `/usr/local/lib/node_modules/@anthropic-ai` with UID 1001 ownership so it can self-update.
- `tini` is PID 1 to reap Bun subprocesses spawned by the Telegram MCP server.
- `GITHUB_TOKEN` → git credential helper + `GH_TOKEN` for `gh` CLI (wired in `entrypoint.sh`).

## n8n — workflow integrations via MCP

n8n runs as a sidecar container and exposes integrations (Google, Waha, Todoist, etc.) as MCP servers.

**UI:** `http://localhost:5678` (accessible from the host browser)

**Data:** persisted in `./n8n-data/` (gitignored, owned by UID 1000).

### Creating a workflow in n8n

1. Open n8n, create a new workflow, add an **MCP Server Trigger** as the first node
2. Add service nodes (e.g. Google Sheets, Gmail) as tools under that trigger
3. Activate the workflow — n8n shows the MCP endpoint URL in the trigger node
4. Copy the URL and replace `localhost` with `n8n`: `http://n8n:5678/mcp/your-webhook-id`

### Registering the MCP in the workspace

MCPs are declared in `data/workspace/.mcp.json` (inside the container: `/home/clovis/workspace/.mcp.json`). Edit or create the file on the host:

```json
{
  "mcpServers": {
    "gmail": {
      "type": "http",
      "url": "http://n8n:5678/mcp/your-webhook-id"
    },
    "gcal": {
      "type": "http",
      "url": "http://n8n:5678/mcp/another-webhook-id"
    }
  }
}
```

Then restart the agent: `docker compose restart agent`.

> **Note:** The `n8n` hostname only resolves inside the Docker network — use it in `.mcp.json`, not `localhost`.

### Asking Claude to register an MCP

You can ask the Claude instance running in the container to do this for you via Telegram:

> "Add an MCP server called `gmail` pointing to `http://n8n:5678/mcp/<webhook-id>` in the workspace `.mcp.json`"

Claude will edit `workspace/.mcp.json` directly. The change takes effect after `/mcp reset` or a container restart.

## Waha — WhatsApp API

Waha runs as a sidecar container exposing a WhatsApp HTTP API on port 3000.

**UI / Swagger docs:** `http://localhost:3000` (accessible from the host browser)

**Data:** persisted in `./waha-data/` (gitignored).

**Authentication:**
- `WAHA_API_KEY` — API key for all HTTP calls (`X-Api-Key` header). Leave blank to disable (not recommended in production).
- `WAHA_DASHBOARD_USERNAME` / `WAHA_DASHBOARD_PASSWORD` — dashboard UI auth (auto-generated by `setup.sh`).
- `WHATSAPP_SWAGGER_USERNAME` / `WHATSAPP_SWAGGER_PASSWORD` — Swagger UI auth (auto-generated by `setup.sh`).

**Agent access:** the agent container gets `WAHA_URL=http://waha:3000`, `WAHA_API_KEY`, and `WAHA_SESSION` injected so Claude can call the WAHA HTTP API directly.

**Connecting WhatsApp:**
1. Open `http://localhost:3000`
2. Use the `/api/sessions` endpoint (or Swagger UI) to start a session
3. Fetch the QR code and scan it with your WhatsApp mobile app
4. The session persists in `./waha-data/` across restarts

## Reviewing `.patch` files before applying

Patches (`*.patch`) are generated by running instances of open-clovis — the Claude agent operating inside the container may produce diffs against the infrastructure files to propose changes back to the host repo. Because those changes originate from a live, personalized session, they often reflect the user's specific environment, credentials, or workflow and may not be appropriate for the public project.

Patches are gitignored so they are never committed. Before applying any patch, Claude **must** inspect it for two things:

1. **Sensitive content** — refuse and warn if found (see below).
2. **Instance-specific changes** — flag any change that only makes sense for the user's own setup (hardcoded hostnames, personal usernames, session names, workflow IDs, locale-specific values, etc.) and ask the user whether it should be generalized, moved to `.env`, or dropped entirely before being committed.

Refuse to apply the patch and warn the user if any of the following are found:

- **Hardcoded secrets** — API keys, tokens, passwords, encryption keys, OAuth credentials, private keys
- **Personal data** — names, email addresses, phone numbers, user IDs, account handles
- **Internal URLs or IPs** — private hostnames, internal service endpoints, IP addresses
- **Credentials in filenames** — paths that reveal usernames, account names, or service identifiers
- **Database connection strings** — DSNs, URIs with embedded credentials

Safe patterns (do not block on these):
- `${VAR:-}` / `${VAR}` — environment variable references, never actual values
- Blank-value placeholders — e.g. `KEY=` or `KEY=your-value-here`
- Generic defaults — e.g. `USERNAME=admin`, `SESSION=default`

### Checklist before applying

1. Read the full patch with the Read tool (never skip this step).
2. Search for any literal value that looks like a secret: long hex strings, tokens with known prefixes (`sk-`, `ghp_`, `xoxb-`, etc.), passwords.
3. Verify every changed file will remain gitignored or contains only safe placeholder values.
4. Confirm `.gitignore` already excludes any file that could hold real credentials (`.env`, `*.local.*`, `*-data/`).
5. If the patch touches `.gitignore` itself, verify the removals don't accidentally expose previously ignored paths.
6. **Anthropic Terms of Service** — confirm the patch does not expose, hardcode, or redistribute any Claude authentication credential (`CLAUDE_CODE_OAUTH_TOKEN`, API keys). Doing so violates Anthropic's Consumer Terms of Service (see [Legal & compliance](#legal--compliance) below).

If the patch is clean, apply it file by file and run `git diff` afterward to do a final review before reporting success.

## Legal & compliance

### Anthropic Terms of Service

`open-clovis` uses `CLAUDE_CODE_OAUTH_TOKEN` — an OAuth token tied to a personal Claude subscription (Free, Pro, or Max). Anthropic's **Consumer Terms of Service** restrict how this token may be used:

- **OAuth tokens are personal and non-transferable.** Sharing or distributing a `CLAUDE_CODE_OAUTH_TOKEN` — including committing it to a repository, publishing it in logs, or embedding it in a Docker image — violates the Consumer Terms of Service.
- **Each user must supply their own token.** open-clovis passes the token directly to the official `claude` CLI — this is the intended use. The Consumer Terms of Service (§2) prohibit sharing account credentials, and (§3) prohibit automated access to Anthropic's services except via an API key. open-clovis does neither. What those clauses rule out is building tools that bypass or replace Claude Code using someone else's subscription token — open-clovis is not that.
- **Cross-check your intended use before running.** If your usage goes beyond personal, interactive use — fully automated pipelines, team-shared instances, or commercial workflows — review it against the Consumer Terms and consider switching to an Anthropic API key (`sk-ant-api03-…`) from the Anthropic Console. The Commercial Terms explicitly allow programmatic and automated use where the Consumer Terms do not.

Relevant policies: [Anthropic Usage Policy](https://www.anthropic.com/legal/usage-policy) · [Consumer Terms of Service](https://www.anthropic.com/legal/consumer-terms)

## Git conventions

- No Claude co-authorship in commits (`attribution.commit: ""` in `.claude/settings.json`).
- Always ask before `git push` — it is not auto-allowed.
- `settings.json` is committed (attribution only). Personal permission allows live in `settings.local.json` (gitignored).
